## 开发目标概述：

实现一个Web App，部署Web App的服务器从另外一个服务器（Nuna服务器）提供的接口下载用户已经采集的音频数据（采集自Nuna，一种录音设备），在服务器上完成ASR等预标注；然后由用户自行选择要用于共享的数据并进行标注，最后将标记结果保存到数据库中，为后续构建音频数据集提供支持。

每个用户都有一个自己的Nuna录音设备，以及一个Nuna账号的userId，基于userId可以从Nuna服务器下载该用户的音频数据。Web App的目的是让用户可以方便地浏览和标注自己的音频数据，然后将标注结果保存下来。

- 平台：一台双核2G的Ubuntu云服务器（有公网ip无域名，公网/本机ip为<SERVER_IP>），峰值200M带宽；本仓库仅部署 app 相关服务，由外部统一提供 gateway 与域名。
- 数据：从 Nuna 服务器下载的音频数据**已经按 1 分钟切片**，每个文件代表一个连续的 60 秒音频片段（下称“片段 / segment”）。片段是标注的最小单位，**块状标注（Block）**必须覆盖**整数个片段**，且片段必须**连续**。**点状标注（Point）**最小也必须覆盖**一个完整片段**（可覆盖多个连续片段，但不得小于 1 个片段）。
- 需求：可以从另一台服务器下载音频数据（API已有），然后用户在平台上登陆后，对自己的音频数据进行打标，需要有图形界面，可以提供例如：
  - 快速回顾：展示本地已经记录的音频（按一分钟一个条目展示），提供时间戳展示、播放（可拖拽进度条）、ASR转文字等功能，还要留出后期让大语言模型/其他语音模型，对音频进行处理和预标注的接口。
  - 打标：用户可以自行选择音频片段（有框可以勾选），然后一起上传，并在上传前提供标注，标注具有可扩展性，即可以比较容易地替换成不同的问题。

## 架构：

代码已经给出大致框架：

- app: web application
- receiver: audio ingest endpoint
- prelabel-service: pre-label service
- runtime: data & environment (not tracked by git)

程序主要分为 app 相关服务（web app / receiver / prelabel）与 runtime 数据目录。gateway 由外部统一提供，本项目只负责把容器端口映射到管理员分配的宿主机端口。数据库以及音频文件均存储在 runtime 目录中，不纳入版本控制，挂载到容器中。

## 详细设计需求：

### app

1. 用户管理（/auth）

   - 注册：基于注册Nuna账号时提供的userId，使用/snippets/s3_audio_downloader.py中get_all_objects，查询该用户的音频列表，若存在则允许注册；注册时设置密码和提供邮箱，将所有注册信息和登陆认证相关的信息，保存到数据库的<用户表>中；
   - 登陆：用户输入userId和密码，进行认证，认证通过后生成session token，保存到cookie中，后续请求均携带该token进行身份验证；
   - 密码重置：用户提供注册时的邮箱，如果相同，则允许重置密码。
2. 数据同步与标注任务展示（/tasks）

   - 界面：登陆后，展示用户的标注任务，登陆后先按“天”展示全部的数据，同时显示一些信息（例如收集的时间、已经标注内容的百分比等等），用户可以点击某一天，进入该天的音频列表页面。
   - 数据同步：每次用户登陆时，系统会自动从Nuna服务器下载该用户最新的音频数据（即当前时间和<用户表>中记录的最后下载时间之间的数据，使用/snippets/s3_audio_downloader.py中download_recent_images即可），然后更新<用户表>中的最后下载时间；下载的音频数据保存到/runtime/audio/{userId}/{date}/目录下，date格式为YYYYMMDD，同时在数据库的<数据表>中保存每个音频文件作为条目，包括用户名、文件路径、时间戳、ASR结果（初始为空）、标注状态（初始为未标注）等信息。
   - 预标注：下载完成后，系统会调用ASR服务（可以使用开源的Whisper模型，或者调用第三方API，例如百度语音识别API）对新下载的音频进行转写，将转写结果保存到对应音频条目的ASR结果字段中；同样也留出接口，方便后续调用大语言模型/其他语音模型进行更复杂的预标注。
   - 界面上展示上一次同步的时间，方便用户了解数据更新情况；同步在后台进行，用户可以继续浏览和标注已有数据；同步完成后，界面刷新，展示最新数据。
3. 音频列表展示与标注（/tasks/{date}）

   - 留出不同标注任务的字页面口，当前版本只实现一种基础标注任务，后续可以扩展SER标注、Ego-Speech标注等；
   - 界面：用户点击某一天后，进入该天的音频列表页面，按分钟展示该天的音频数据（但可以按每小时折叠成一块，每15分钟再折叠在一块，可以勾选一个块来全选当前全部片段，也可以跨块勾选），每个条目展示时间戳、播放控件（可拖拽进度条）、ASR结果（如果有）等信息；用户可以选择多个音频条目进行批量标注。
   - 播放功能：每个音频条目提供播放功能，用户可以点击播放按钮，播放该音频片段，支持暂停、继续和拖拽进度条。
   - 标注功能：
     - 先进行块状标注：用户选择多个音频条目后，点击“标注”按钮，弹出标注界面，用户可以针对选中的音频进行标注；标注内容可以是文本输入框，也可以是预定义的选项（例如情感分类、关键词等），具体标注问题可以在后续版本中扩展和修改。
     - 然后进行点状标注；完成块状标注后，用户可以再从多个音频条目中多次点标注，即选择一个音频，进行细粒度的点状标注，标注内容同样可以在后续版本中扩展和修改。
     - 提交标注：用户完成标注后，点击“提交”按钮，将标注结果保存到数据库的<数据表>中，同时更新对应音频条目的标注状态为“已标注”。已经标注的结果可以在界面上查看和修改。
   - 删除功能：用户可以选择一个或多个音频条目，点击“删除”按钮，将这些音频文件从服务器上删除，同时从数据库中删除对应的条目；删除操作需要二次确认，防止误操作；删除的音频文件无法恢复，也不会被从Nuna服务器重新下载。

### 数据库设计

设计一个SQLite数据库，包含以下主要表：

1. 用户表（users）
   - userId: 用户的Nuna账号ID，主键
   - password_hash: 用户密码的哈希值
   - email: 用户注册时提供的邮箱
   - last_sync_time: 上次从Nuna服务器同步数据的时间戳
2. 数据表（audio_data）
   - id: 主键，自增
   - userId: 外键，关联用户表
   - file_path: 音频文件在服务器上的存储路径
   - timestamp: 音频文件的时间戳
   - asr_result: 预标注的ASR转写结果
   - annotation: 用户标注的结果，JSON格式存储
   - status: 标注状态（未标注、已标注）

## 其他设计考虑

- 需要注意Docker的部署，Docker Compose 固化部署，配置全部外置化：12-Factor 风格，数据与状态的迁移策略，权限与隔离：以“app未来在共享内网机器运行”作为约束来设计，网络与端口：先按“未来只有内网可达”设计等等等。一切以方便迁移、且迁移到学校公用内网服务器上能运行为主
- 音频文件最好放在宿主机目录下，再挂载到app与db的docker
  - db可以采用sqlite，这样省事，不用额外的docker，和app放在一起即可；但是最好开发的时候留出替换接口，方便后续数据库扩大使用mysql这些需要放到docker里的大型数据库——用环境变量配置数据库、文件存储与 DB 解耦等（DB只存引用与元数据）
- 前端后端尽可能使用现成框架和比较简单的技术栈，方便新手入门以及后续维护
- 代码框架已经给出大致结构，后续开发时在此基础上进行扩展和完善；对于部分文件夹给出的站位文件和站位代码，可以根据实际需求进行修改和替换；部分功能的实现需要用到定制代码，参考/snippets/目录下的代码片段
  - /app：存放web app和数据库相关代码，只存放代码与dockerfile等配置文件
  - /runtime：存放运行时数据和环境配置，不纳入版本控制；所有配置文件以及api密钥等均放在这里
  - /snippets：存放一些代码片段，供开发时参考和使用
   - /compose：按服务拆分 compose 文件，并通过 stack.env 统一管理宿主机端口映射
   - /scripts：存放辅助脚本，例如一键初始化、一键部署等
