# 数据预标注系统（MVP）开发需求说明

## 一、目标（Objective）

在现有数据标注网站基础上，新增“预标注（Pre-labeling）”能力，实现：

* 对已有数据按“天”为单位触发预标注任务
* 自动生成预标注结果（由独立服务完成）
* 在现有 UI 中可视化展示预标注结果
* 支持后续人机协同标注（本阶段先不实现）

⚠️ 当前阶段目标：

> **打通完整链路（任务触发 → 执行 → 存储 → 可视化）**
> 不涉及模型优化、复杂 pipeline、训练闭环

---

# 二、总体架构（必须遵守）

## 架构原则（核心约束）

1. **UI 与主网站融合**
2. **预标注计算逻辑完全解耦（独立服务）**
3. **原始数据只读**
4. **预标注结果与正式标注完全隔离（数据库层）**
5. **所有预标注任务必须异步执行**

---

## 系统拆分

### 1️⃣ 主网站（已有 docker）

负责：

* UI 展示
* 用户交互
* 任务发起
* 状态查询
* 结果展示

❗禁止：

* 写入预标注核心逻辑
* 执行模型推理
* 做重计算任务

---

### 2️⃣ 预标注服务（新建 docker）

负责：

* 执行预标注 pipeline
* 调用外部 API（ASR / SED 等）
* 数据处理（分钟级切分等）
* 写入预标注结果
* 任务状态管理

---

### 3️⃣ 数据库（共享）

* 原始数据表（只读，或仅修改“预标注状态”字段）
* 正式标注表（不修改）
* 新增预标注相关表（本需求重点）

---

# 三、数据与系统解耦要求（必须严格执行）

## 1. 数据共享原则

| 数据类型  | 访问方式          | 权限       |
| ----- | ------------- | -------- |
| 原始数据  | 主网站 + 预标注服务共享 | 只读，仅能写状态       |
| 正式标注  | 主网站读写         | 预标注服务禁止写 |
| 预标注结果 | 预标注服务写 / 主网站读 | 独立表      |

---

## 2. 代码解耦原则

### 主网站：

* ❗ 不包含任何预标注算法逻辑
* ❗ 不包含音频/视频处理逻辑
* 仅做 API 调用

### 预标注服务：

* ❗ 不包含 UI 逻辑
* ❗ 不包含用户权限逻辑
* 仅提供 API + worker

---

## 3. 服务通信方式

采用 HTTP API（MVP 阶段）

主网站 → 预标注服务：

* 创建任务
* 查询任务状态

---

状态：

### ② 任务状态展示

展示内容：

* 开始时间 / 结束时间
* 是否已有历史结果

---

### ③ 预标注入口

点击后进入：

👉 新页面：`/tasks/{day_id}/prelabel`

---

## 4.2 预标注结果页面（新页面）

功能：

* 展示该天所有预标注结果
* 按“分钟粒度”展示（整理风格和正式标注ui类似）
* 可切换不同任务类型（主要分为ASR和SED&ASC&HAR两类）
  * ASR按照每分钟展示文本结果
  * SED&ASC&HAR按照正式标注种中的块标注和点标注展示方法

MVP 要求：
* 不要求复杂交互
* 不要求编辑功能（后续再做）
* 目前预标注模块还未开发，留出接口对接

---

## 4.3 后端接口（主网站）

### 1. 创建预标注任务

```
POST /api/prelabel/create
```

参数：

* day_id

逻辑：

* 创建 job（写 DB）
* 调用预标注服务 API
* 返回 job_id

---

### 2. 查询任务状态

```
GET /api/prelabel/status/{job_id}
```

---

### 3. 获取预标注结果

```
GET /api/prelabel/results?day_id=xxx
```

---

## 4.4 权限与限制（MVP 必须加）

* 同一天只允许一个 active job
* 已完成任务默认不重复运行

---

# 五、预标注服务（新 docker）开发需求

## 5.1 服务职责

必须实现：

* 接收任务请求
* 拉取原始数据
* 执行 pipeline（可先 mock）
* 写入数据库
* 更新任务状态

---

## 5.2 API 设计

### 1. 创建任务

```
POST /prelabel/run
```

参数：

* job_id
* day_id

---

### 2. 查询状态

```
GET /prelabel/status/{job_id}
```

---

## 5.3 执行逻辑（MVP）

流程：

1. 根据 day_id 查询原始数据
2. 遍历数据（按分钟）
3. 执行预标注逻辑（可先 mock）
4. 写入结果表
5. 更新 job 状态

---

## 5.4 异步执行（必须）

不能同步执行任务。

建议：

* 简单方案：后台线程 / asyncio
* 推荐：worker（如 Celery / queue）

---

## 5.5 日志与容错

必须记录：

* 每个 job 日志
* 错误信息
* 失败状态

---

# 七、关键约束（必须遵守）

## 1. 不污染正式标注

* ❗预标注结果绝不能写入正式标注表

---

## 2. 原始数据只读

* ❗预标注服务禁止修改原始数据，除了状态

---

## 3. 不同步阻塞 UI

* ❗任务必须异步执行

---

## 4. 可重复运行

* 支持同一天多次跑（不同 job）

---

## 5. 结果与版本可追溯

* 每次运行必须记录 job_id

# 十、一句话总结架构

> **预标注是主网站的一个功能模块，但它的计算执行必须是一个独立服务；
